## SECTION 2: Purpose

Repository refinement involves structural modifications that carry significant risk to historical traceability and architectural integrity. Therefore, an execution strategy must exist before any physical refinement begins.

This execution strategy permanently defines *how* repository refinement will be performed. It guarantees that all actions are controlled, strictly validated, fully documented, and 100% reversible, preventing accidental knowledge loss during cleanup.

---

## SECTION 3: Guiding Principles

The execution of repository refinement is governed by the following constitutional principles:

* **Knowledge First**: No action may proceed until the knowledge contained within an asset is fully understood and classified.
* **Evidence Before Action**: Duplication or obsolescence must be proven before any action is approved.
* **One Approved Change At A Time**: Refinement occurs in discrete, atomic steps to prevent cascading failures.
* **No Knowledge Loss**: Deletions or merges must perfectly preserve the underlying logic and context.
* **Repository Integrity**: Modifications must not break Markdown links, test dependencies, or architectural dependencies.
* **Architecture Preservation**: Refinement alters organization, never the foundational rules of the system.
* **Governance First**: Process and documentation always precede implementation.
* **Deterministic Preservation**: Refinement must never alter the output of any astrological calculation.

---

## SECTION 4: Repository Refinement Workflow

The execution of a refinement action must follow a rigid, cyclical workflow:

1. **Inventory Verification**: Confirm the target asset exists in the Master Inventory.
2. **Classification Verification**: Confirm the asset's approved classification.
3. **Dependency Verification**: Review the Dependency and Impact Analysis for downstream risks.
4. **Decision Register Review**: Ensure a formal refinement proposal exists.
5. **Architecture Approval**: Secure formal approval in the Decision Register.
6. **Controlled Refinement**: Execute the single, approved file operation (move, rename, merge).
7. **Repository Validation**: Verify that no links or dependencies were broken by the action.
8. **Git Checkpoint**: Commit the stable, refined state.
9. **Documentation Update**: Mark the Decision Register entry as "Implemented".
10. **Repeat**: Return to step 1 for the next action.

---

## SECTION 5: Execution Phases

The overall GM-005 milestone execution is divided into the following phases:

* **Phase 1: Repository Analysis**
  * *Objective*: Catalog, classify, and analyze all assets for duplication and dependency impact.
  * *Completion Criteria*: All five foundation analysis documents are approved.
* **Phase 2: Decision Approval**
  * *Objective*: Populate the Decision Register with evidence-based refinement proposals.
  * *Completion Criteria*: Refinement proposals are reviewed and formally marked "Approved".
* **Phase 3: Controlled Refinement**
  * *Objective*: Execute the approved physical changes to the repository file system.
  * *Completion Criteria*: Only approved operations are performed; zero unapproved changes exist.
* **Phase 4: Validation**
  * *Objective*: Test the repository structure for broken links, missing files, or severed dependencies.
  * *Completion Criteria*: 100% dependency integrity is confirmed.
* **Phase 5: Checkpoint**
  * *Objective*: Secure the refined repository state in source control.
  * *Completion Criteria*: Git commits are finalized with clear tracebacks to the Decision Register.
* **Phase 6: Final Verification**
  * *Objective*: Confirm the repository is ready for the Version 2 Foundation (GM-008).
  * *Completion Criteria*: The GM-005 Completion Report is drafted and approved.

## SECTION 6: Approval Workflow

Implementation is strictly prohibited until the following sequence of approvals is achieved:

1. **Architectural Review**: The proposed action does not violate Golden Master architecture.
2. **Evidence Verification**: The duplication or obsolescence claim is factually accurate.
3. **Dependency Review**: All indirect and direct dependencies have been mapped and mitigated.
4. **Risk Review**: The risk of knowledge loss has been evaluated as zero or perfectly mitigated.
5. **Decision Register Approval**: The status is officially flipped to "Approved" in `docs/governance/REPOSITORY_REFINEMENT_DECISION_REGISTER_v1.0.md`.

---

## SECTION 7: Validation Workflow

After every refinement action, the repository must be immediately verified to ensure integrity:

* **Repository consistency**: No broken internal file paths or orphaned directories.
* **Knowledge preservation**: A manual spot-check ensures merged documents retained all critical context.
* **Reference integrity**: Markdown hyperlinks and code imports correctly resolve to the new locations.
* **Architecture consistency**: The structural changes align with the Golden Master Manifest.
* **Governance consistency**: No constitutional rules or registries were accidentally archived.
* **Documentation consistency**: The Master Inventory is conceptually up-to-date with the new state.

---

## SECTION 8: Git Checkpoint Strategy

Git checkpoints (commits or tags) represent stable, validated repository states. Checkpoints must occur at the following intervals:

* **After stable analysis package**: Once the GM-005 analysis documents are drafted.
* **After approved refinement package**: Before any physical changes are executed, securing a rollback point.
* **After verified consolidation package**: Following the execution and validation of a batch of approved decisions.
* **After milestone completion**: A final tag signifying the closure of GM-005.

---

## SECTION 9: Rollback Strategy

If a refinement action breaks dependency integrity or results in knowledge loss, the following rollback principles apply:

* **Git History**: The primary mechanism for reverting an unstable physical change.
* **Decision Register Traceability**: If an action is rolled back, its entry in the Decision Register is updated to "Rejected" or "Deferred" with explanatory notes.
* **Documentation Traceability**: Reverting the repository restores the exact state defined in the initial Master Inventory.
* **Reversible Refinement**: Refinement must always be executed in a way that allows 100% clean reversal.
* **Knowledge Preservation**: Restoring the pre-refinement state guarantees no knowledge remains lost.

Repository refinement for milestone GM-005 is officially complete only when:
