# ADR-016: Deterministic Communication of System Warnings

## ADR ID
ADR-016

## Title
Deterministic Communication of System Warnings

## Status
Accepted

## Recovery Classification

## Context
When system degradation occurs (e.g., missing transit data), the frontend needs to be aware of the missing data. A method was required to communicate these failures to the UI without relying on the LLM to explain the issue.

## Repository Evidence
"- I recommended the **Deterministic** approach (Alternative B) because relying on the LLM to communicate system failures invites hallucination and violates our mandate for explicit determinism."

## Decision
Missing data warnings and system failures (e.g., "Transit Data Unavailable") must be appended and communicated deterministically rather than generated or interpreted by the LLM.

## Architectural Rationale
This fixes a UI edge case while preserving strict prompt-bounding logic. Relying on the LLM to communicate system failures invites hallucination and violates the mandate for explicit determinism.

## Consequences
The UI explicitly handles warning payloads deterministically. The LLM is bypassed for communicating internal system health or missing engine signals.
