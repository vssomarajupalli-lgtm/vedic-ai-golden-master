## Context
During Phase 9, there was a critical need to ensure the calculation engines remained mathematically isolated from specific formula logic to prevent the backend from becoming an unmaintainable monolith.

## Repository Evidence
"`FORMULA_REPOSITORY_DESIGN` strictly forbids hardcoded astrology rules inside Python execution code, pushing rule evaluation to an externalized format."

## Decision
Astrological rules (e.g., "Jupiter in the 7th House dictates X outcome") shall never be hardcoded into Python engine classes. They shall be pushed to externalized declarative formats (YAML/JSON).

## Architectural Rationale
This guarantees that the Python engines only perform generic math (calculating planets, houses, dashas) while the actual astrological logic is dynamically injected via configurations. It preserves deterministic execution and enforces a strict boundary between execution and astrological domain knowledge.

## Consequences
Python engine code is vastly simplified and isolated. Astrological logic is maintained exclusively by domain experts editing external files without touching source code.
